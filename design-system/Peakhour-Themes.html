My take · the architecture

Themes as data, scheduled by occasion & country.

Because every component reads the same shadcn token slots, a theme is just a small CSS file — a row in a database. That's the unlock: the CMS doesn't redesign anything, it swaps the active token set. Diwali in Mumbai, Christmas in London, a national-day palette in Riyadh — all the same mechanism, scheduled ahead of time.

/* a theme = one drop-in token file */
[data-theme="diwali"] {
  --brand: oklch(.78 .16 72); /* marigold */
  --background: oklch(.2 .035 335);
  --primary: var(--brand); …
}

// CMS schedule (proposed 3 months ahead)
{ region:"IN", theme:"diwali", start:"2026-10-20" }
1

Token-only themes

Each theme ships as tokens/theme-<name>.css. Zero component changes — copy a block from this page and it's production code.

2

Scheduled by calendar & country

A theme registry maps region + date window → active theme. The CMS sets data-theme at request time.

3

Proposed ahead of time

Claude generates festival palettes from a calendar of events per country and opens them for review ~3 months out — never a last-minute scramble.

4

Reviewed, then auto-published

Approved themes go live on schedule and retire automatically. Every theme stays on-system because it can only touch the token layer.
